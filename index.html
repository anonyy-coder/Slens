Motivation

Two structural failure modes that collapse skills toward zero.

A naive design — run a fixed task with the skill enabled and report the resulting success rate — breaks down for two structural reasons. SkillTestBench is engineered to neutralize both.

Failure mode 1

Baseline conflation.

Absolute success rate confounds the marginal contribution of the skill with the underlying competence of the agent. A skill that reaches 78 % on a task may represent only a 3-point improvement over a 75 % no-skill baseline.

Fix. Paired execution. Every task is run twice — with the skill mounted and without — holding agent, task, and seed fixed.
Failure mode 2

Difficulty misalignment.

Tasks too easy (baseline near 1) or too hard (baseline near 0) leave little room for a skill to reveal value. An evaluation that mixes such cases indiscriminately collapses most skills toward U ≈ 0 even when they are useful.

Fix. Baseline-calibrated, difficulty-aligned task generation, seeded from each skill's own profile rather than a fixed suite.
The framework

Two deployment-facing axes, one execution pass.

Each containerized run produces a single evidence bundle — verifier reward, execution trajectory, file-system delta, and outbound network log. Utility reports two sub-dimensions independently and does not combine them; Safety composes static existence with dynamic exploitability into a 100-based deduction score.

Axis 1. Utility

value over baseline

The skill's value relative to a no-skill baseline. Reported as two independent sub-dimensions so capability gains and resource savings are read separately, never silently averaged.

1a. Effectivenesspass_rate_gain

Proportion of judge items the skill additionally unlocks beyond baseline, measured per scenario across 5–6 binary judge items.

pass_rate_gain = 0 if wi_passed = 0 0 if 0 < wi_passed ≤ wo_passed (wi_passed - wo_passed) / total_items otherwise
Absolute item-count differences over total items — not the relative form (wi-wo)/(1-wo) — keep the metric stable across baseline regimes when judge items are discrete and few.
1b. Efficiencyefficiency_score

Resource savings on tasks where both wi and wo runs completed at least one item. Time and effective tokens (input minus cache) contribute equally.

time_score = clamp((wo_time - wi_time) / wo_time, 0, 1) token_score = clamp((wo_tok - wi_tok) / wo_tok, 0, 1) efficiency_score = (time_score + token_score) / 2
Reported separately from effectiveness. A skill can lift task completion without saving resources, and vice versa — both are deployment-relevant.

Axis 2. Safety

risk under runtime evidence

A 100-based score derived per skill: each static finding has an existence probability, each dynamic test produces an exploitability probability, and the product weights a base deduction drawn from the finding's severity tier.

2a. Existence (static)existence_confidence

Static scanner output answering: does this risk objectively exist in the skill code? Range [0, 1] with calibrated tiers (1.0 unambiguous → 0.2 likely false positive).

2b. Exploitability (dynamic)exploitability

LLM Security Judge output combining run trajectory, verifier rule checks, and the static finding's predicted trigger path. Five-tier verdict from confirmed (≥ 0.85, end-to-end attack path) down to likely_false_positive.

Scoresecurity_score
deduction_i = base_score(severity) × existence_i × exploitability_i security_score = max(10, 100 - Σ_i deduction_i)
High15
Med10
Low5
Probability semantics: P(risk) = P(exists) × P(exploitable | exists). Floor of 10 avoids negative-score ambiguity; defaults of 0.6 apply when a measurement is absent.

Pipeline

From skill package to scored evidence.

Given K = (D, F, M) — documentation, files, metadata — SkillTestBench compiles each skill into verifier-backed Harbor tasks and emits a single evidence bundle from which both axes are computed.

  1. 1

    Profile

    Static scan of SKILL.md, scripts, and metadata to extract claimed capabilities and risk findings.

  2. 2

    Generate

    Capability-targeted scenarios (U1 / U2 / U3) with 5–6 judge items each, calibrated against the no-skill baseline.

  3. 3

    Execute

    Containerized Harbor runs, paired wi / wo with matched seeds. Sandbox records FS diff and outbound network in real time.

  4. 4

    Judge

    LLM judge scores binary item completion; security judge fuses verifier rule checks with semantic analysis.

  5. 5

    Report

    Per-skill skill_report.json with pass_rate_gain, efficiency_score, and security.score.

In-the-wild discovery

SkillTestBench at the point of adoption.

A Chrome MV3 extension recognizes skill-format repositories on GitHub by their SKILL.md structure, queries the evaluation server, and renders a compact two-axis panel directly on the repository page — before the developer installs anything.

  1. Open chrome://extensions. Enable Developer mode in the top-right corner.
  2. Click Load unpacked. Select the chrome_extension/ directory from the source tree.
  3. Start the local evaluation server. Run python local_mock_server/mock_skill_server.py on port 8765.
  4. Visit any skill repository on GitHub. The two-axis panel injects automatically wherever a SKILL.md is detected.
