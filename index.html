04 / How it works

A skill becomes a verdict in five steps.

Given K = (D, F, M) — documentation, files, and metadata — every skill passes through the same pipeline. The same execution pass produces both axes.

01 Profile Static scan of SKILL.md, scripts, dependencies. Extracts claimed capabilities and risk findings with existence_confidence ∈ [0, 1]. → static_scan.json
02 Generate Capability-targeted scenarios (U1 / U2 / U3) with 5–6 binary judge items each, calibrated against the no-skill baseline difficulty. → scenarios/U*.yaml
03 Execute Containerized Harbor runs, paired wi / wo with matched seeds. Sandbox records FS diff and outbound network traffic in real time. → fs.diff · net.log
04 Judge LLM judge scores each binary item; security judge composes existence × exploitability against the runtime trace. → judges/*.json
05 Report Per-skill skill_report.json with pass_rate_gain, efficiency_score, and security.score = max(10, 100 - Σ base × existence × exploit). → skill_report.json
